⚡ Core CLI Commands
🚀
Init & Plan
$terraform initcopy
Download providers, initialize backend, set up modules
$terraform init -upgradecopy
Upgrade provider versions to latest allowed by constraints
$terraform plancopy
Show what changes will be made — dry run, no side effects
$terraform plan -out=tfplancopy
Save plan to file — apply this exact plan later (CI/CD pattern)
$terraform plan -var-file=prod.tfvarscopy
Use a specific variable file for environment overrides
⚙️
Apply & Destroy
$terraform applycopy
Apply changes — prompts for confirmation
$terraform apply -auto-approvecopy
Skip confirmation — use in CI/CD pipelines only
$terraform apply tfplancopy
Apply a saved plan file — deterministic, no surprises
$terraform destroycopy
Destroy all managed infrastructure
$terraform destroy -target=aws_instance.webcopy
Destroy a specific resource only

📝 HCL Syntax
🏗️
Core Blocks
# Provider configuration terraform { required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" } } backend "s3" { bucket = "my-tf-state" key = "prod/terraform.tfstate" region = "us-east-1" } } provider "aws" { region = var.region } # Resource block resource "aws_instance" "web" { ami = data.aws_ami.ubuntu.id instance_type = var.instance_type tags = { Name = "web-server" Env = var.environment } }
📦
Variables & Outputs
# Input variable variable "instance_type" { description = "EC2 instance size" type = string default = "t3.micro" validation { condition = contains(["t3.micro", "t3.small"], var.instance_type) error_message = "Must be t3.micro or t3.small." } } # Local value - computed, not input locals { name_prefix = "${var.env}-${var.app}" common_tags = { ManagedBy = "terraform" Env = var.env } } # Output value output "instance_ip" { description = "Public IP of web server" value = aws_instance.web.public_ip sensitive = false }
🔄
Meta-arguments & Loops
# count - create N identical resources resource "aws_instance" "workers" { count = 3 instance_type = "t3.micro" ami = var.ami_id tags = { Name = "worker-${count.index}" } } # for_each - create resources from a map resource "aws_s3_bucket" "buckets" { for_each = toset(["logs", "data", "backups"]) bucket = "my-app-${each.key}" } # depends_on - explicit dependency resource "aws_db_instance" "db" { depends_on = [aws_security_group.db_sg] } # lifecycle rules resource "aws_instance" "web" { lifecycle { create_before_destroy = true prevent_destroy = true ignore_changes = [ami] } }
🗄️
Data Sources
# Read existing infra not managed by Terraform data "aws_ami" "ubuntu" { most_recent = true owners = ["099720109477"] # Canonical filter { name = "name" values = ["ubuntu/images/hvm-ssd/ubuntu-*"] } } # Reference it in a resource resource "aws_instance" "web" { ami = data.aws_ami.ubuntu.id } # Read from remote state (cross-stack reference) data "terraform_remote_state" "vpc" { backend = "s3" config = { bucket = "my-tf-state" key = "vpc/terraform.tfstate" region = "us-east-1" } } # Use: data.terraform_remote_state.vpc.outputs.vpc_id
💾 State Management
🗂️
State Commands
$terraform state listcopy
List all resources tracked in state
$terraform state show aws_instance.webcopy
Show full details of one resource in state
$terraform state mv aws_instance.old aws_instance.newcopy
Rename a resource in state without destroying it
$terraform state rm aws_instance.webcopy
Remove resource from state (resource still exists in cloud)
$terraform import aws_instance.web i-1234567890copy
Import existing cloud resource into Terraform state
$terraform refreshcopy
Sync state with real-world cloud resources
🔒
Remote State & Locking
Remote Backend
Store .tfstate in S3, GCS, or Terraform Cloud — never commit state to git
State Locking
DynamoDB table prevents concurrent applies from corrupting state
State Encryption
S3 SSE-KMS encrypts state at rest — state contains secrets!
Workspaces
Separate state per env (dev/staging/prod) in same codebase
$terraform workspace new stagingcopy
Create a new workspace for environment isolation
$terraform workspace select prodcopy
Switch to production workspace
$terraform force-unlock LOCK_IDcopy
Release a stuck state lock (use carefully)

Module Structure
# modules/vpc/main.tf variable "cidr_block" { type = string } variable "env" { type = string } resource "aws_vpc" "this" { cidr_block = var.cidr_block tags = { Name = "${var.env}-vpc" } } output "vpc_id" { value = aws_vpc.this.id } # root/main.tf — calling the module module "vpc" { source = "./modules/vpc" cidr_block = "10.0.0.0/16" env = var.environment } # Reference module output resource "aws_subnet" "public" { vpc_id = module.vpc.vpc_id }
🌐
Module Sources
# Local path source = "./modules/vpc" # Terraform Registry (public) source = "terraform-aws-modules/vpc/aws" version = "~> 5.0" # Private Git repo source = "git::https://github.com/org/modules.git//vpc?ref=v2.1.0" # S3 bucket source = "s3::https://s3.amazonaws.com/bucket/modules/vpc.zip"
DRY Principle
Modules eliminate copy-paste across environments — one source of truth
Version Pinning
Always pin module versions in production to prevent surprise changes
🎯 Senior Interview Talking Points
💡
Key Concepts to Know Cold
Idempotency
Running terraform apply twice produces the same result — desired state model
Drift Detection
State diverges from reality when infra changes outside Terraform — terraform refresh fixes this
Plan vs Apply
Plan = git diff for infra. Always review plan before apply in prod
Sensitive Values
Mark outputs sensitive=true, use Vault provider or AWS Secrets Manager — never hardcode secrets
count vs for_each
for_each is preferred — count causes index shift issues when middle items are removed
Provider Locking
.terraform.lock.hcl pins provider versions — commit this file to git
🏆
CI/CD Pipeline Pattern
# Typical CI/CD workflow (GitHub Actions / GitLab CI) # PR opened: terraform fmt -check # fail if unformatted terraform validate # syntax check terraform plan -out=plan # post plan to PR comment # PR merged to main: terraform apply plan # apply the exact saved plan # Key CI/CD env vars: TF_VAR_environment = "prod" TF_CLI_ARGS_plan = "-lock-timeout=60s" AWS_ROLE_ARN = "arn:aws:iam::123:role/tf-deploy"
OIDC over long-lived keys
Use GitHub OIDC → AWS IAM role federation instead of storing AWS_ACCESS_KEY in CI secrets — no static credentials
⚠️
Common Gotchas
Never edit state manually
Use terraform state mv/rm — manual edits corrupt the JSON and break everything
Destroy ordering
depends_on controls creation order but also reverse-destruction order — matters for IAM/network resources
prevent_destroy
Add to databases and stateful resources in prod — stops accidental terraform destroy
Partial apply
If apply fails midway, state reflects partial changes — re-running apply is safe due to idempotency
Large refactors
Use moved {} blocks (TF 1.1+) when renaming resources — avoids destroy/recreate
State in git = bad
tfstate contains plaintext secrets. Always use remote backend with encryption
